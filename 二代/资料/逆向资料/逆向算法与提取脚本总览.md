# 逆向算法与提取脚本总览

> 这一份是面向“以后还要继续自己出数据、修数据、扩数据”的索引。和 `复刻可用数据清单.md` 不同，那里偏结果；这里偏**方法**。

## 为什么算法也必须保留

如果只保留导出的 JSON、PNG、CSV，那么你以后可以：

- 做复刻原型
- 搭地图和港口系统
- 用现成资源做占位

但你很难继续：

- 重跑新的资源提取
- 修正已有字段的误判
- 从未解完的文件里继续深挖
- 验证某个资源到底是不是原始结果还是 demo 派生值
- 把流程迁移到你自己的工程里

所以对长期价值来说，**脚本和格式说明与结果数据同样重要**。

## 脚本分层看法

建议把这些脚本按 5 类理解。

### 1. 容器解压层
核心目标：先把 `.lzw` / LS11 / LS10 / Ls12 容器拆开。

关键脚本：
- `scripts/ls11_decode.py`
- `scripts/inventory_lzw.py`

这是最底层、最基础的一层。只要这层稳定，后面的大部分资源提取都能继续做。

### 2. 图像渲染层
核心目标：把拆出来的二进制 part 按正确像素编码还原成图。

关键脚本：
- `scripts/render_kao_v4.py`
- `scripts/render_disc_v1.py`
- `scripts/render_portchip_v2.py`
- `scripts/render_portmap_v2.py`
- `scripts/render_worldmap_v1.py`
- `scripts/render_worldmap_v2.py`
- `scripts/render_char_v2.py`
- `scripts/render_iap_v1.py`
- `scripts/render_font_v1.py`
- `scripts/render_worldmap_hires.py`

对复刻来说，这一层的重要性非常高，因为它决定你能不能持续生产：
- 头像
- 道具图标
- 港口图
- 世界图
- 决斗 sprite
- 行走 sprite

### 3. 数据表提取层
核心目标：从 `.dat` / `Data1.lzw` 里提取结构化数据。

关键脚本：
- `scripts/extract_dat_phase1.py`
- `scripts/extract_data1_parts.py`
- `scripts/extract_remaining_dats.py`
- `scripts/extract_messages.py`
- `scripts/extract_ships.py`
- `scripts/decode_text.py`

这层直接关系到：
- 港口底层数值
- 风向 / 洋流网格
- 海怪数据
- 船只数据
- 文本与消息

### 4. 事件 / CG / 反汇编层
核心目标：继续突破还没完全解完的事件图、文本、脚本结构。

关键脚本：
- `scripts/decode_eventcg.py`
- `scripts/decode_eventcg_vram.py`
- `scripts/disasm_exe.py`
- `scripts/disasm_joao_control_layer.py`
- `scripts/scan_ls11_in_exe.py`
- `scripts/sndt_disasm.py`
- `scripts/sndt_trace_runner.py`
- `docs/archive/REVERSE_ENGINEERING_NOTES.md`
- `docs/main_exe_decoder_disasm.txt`

这是未来继续深挖剧情系统、事件系统和演出系统的关键层。

### 5. 拓扑 / 剧情分析层
核心目标：把文本和脚本结构变成人能读、游戏能设计的结构图。

关键脚本：
- `scripts/export_event_overlap_graph.py`
- `scripts/export_event_script_bridge.py`
- `scripts/export_event_text_index.py`
- `scripts/export_game_topology_bundle.py`
- `scripts/export_line_bridge_matrix.py`
- `scripts/export_overlap_bridge_table.py`
- `scripts/export_sndt_script_index.py`
- `scripts/export_sndt_topology.py`
- `scripts/analyze_*`

这一层更偏“设计资料”和“剧情系统建模”，对你以后搭事件状态机非常重要。

## 最值得优先掌握的 8 个脚本

如果以后你要自己接着做，我建议先熟这 8 个：

1. `scripts/ls11_decode.py`
2. `scripts/inventory_lzw.py`
3. `scripts/render_portchip_v2.py`
4. `scripts/render_portmap_v2.py`
5. `scripts/render_worldmap_v2.py`
6. `scripts/extract_dat_phase1.py`
7. `scripts/decode_text.py`

原因很简单：

- 第 8 个解决“事件 CG 这个最大难点怎么继续攻”

## 这些脚本现在能不能直接跑？

要分三种情况看：

### A. 逻辑本身有效，但不能直接在本仓库运行
原因通常是：
- 仍引用原作者本机路径，如 `/Users/dong/Projects/Koukai2`
- 依赖原始游戏文件未随仓库导入

典型脚本：
- `inventory_lzw.py`
- `decode_text.py`
- `extract_dat_phase1.py`
- `decode_eventcg.py`

### B. 逻辑和路径都相对容易改造
只要把输入路径改成你自己的原始资源目录，通常就能继续用。

典型脚本：
- `ls11_decode.py`
- 多数 `render_*.py`

### C. 仍处在研究 / 攻坚阶段
这些脚本更像“研究工作台”，不是稳定产线工具。

典型脚本：
- `decode_eventcg.py`
- `sndt_disasm.py`
- `analyze_*`
- `experiments/*`

## 怎么用这些脚本服务复刻

### 用法 1：当“可重复生成器”
不要只把 PNG 当最终结果，而是把脚本当成生成器。

例如：
- 港口图坏了，可以重跑
- 想换命名方式，可以重导
- 想做高清或批处理，可以改脚本

### 用法 2：当“格式参考实现”
即使以后你不用 Python，也可以把这里的实现迁到：
- Rust
- C# / Unity
- Godot GDScript
- TypeScript 工具链

### 用法 3：当“研究证据链”
以后你如果要判断一条数据是否可信，最有用的不是看最后的 JSON，而是回到：
- 是哪个脚本出的
- 用了哪种解码假设
- 原始输入是什么
- 哪一步仍是推测

## 导入边界

这一轮我做的是：
- **完整导入 `scripts/` 树**，包括 `experiments/`
- 导入算法 / 反汇编相关关键文档
- 不改原作者脚本内容本身，只把它们作为“来源层”保留

这意味着：
- 现在最重要的是“留住方法”
- 不是立刻把所有脚本都改到可运行
- 真要落到你自己的复刻工程时，再挑核心脚本做一次“可移植化改造”更合理

## 下一步如果要继续补

最值得继续做的两件事：

1. **给脚本加一份“可移植改造清单”**
   - 哪些 absolute path 要替换
   - 哪些依赖要安装
   - 哪些输入文件必须准备

2. **把核心算法再提炼成你自己的最小工具包**
   - 先只保留 LS11 解压
   - 港口图渲染
   - 世界图渲染
   - 文本解码
   - 港口数值提取

这样以后复刻工程就不用背整个研究仓库的复杂性。
