# 脚本可移植改造清单

> 目标不是立刻把 `sea2_demo` 的 58 个脚本全部改成能跑，而是把它们从“作者本机研究环境”转成“你自己的可维护工具链”。

## 0. 当前判断

目前导入的脚本已经足够做《大航海时代 II》复刻的数据与资源再生产，但它们还带着明显的研究仓特征：

- 存在作者本机绝对路径
- 一部分脚本依赖原始 DOS 版资源目录
- 一部分脚本是稳定产线工具
- 一部分脚本只是研究 / 反汇编工作台

因此最合理的迁移方法不是“大改全部”，而是：

1. 先把**核心 5~8 个脚本**迁成稳定工具
2. 再把中等价值脚本纳入研究工具集
3. 最后保留 experiments 仅作历史参考

## 2. 推荐的目标目录结构

建议你未来的复刻工程或工具仓统一采用一个与原作者机器无关的输入布局，例如：

```text
uw2-toolkit/
  raw/
    Koukai2/
      Main.exe
      Chip_no.dat
      KoeiCht.txt
      Message.dat
      1.pat
      2.pat
      *.lzw
      *.dat
  output/
  toolkit/
  scripts/
```

核心原则：

- **原始输入固定放在 `raw/Koukai2/`**
- **所有输出固定写到 `output/`**
- **算法代码放在 `toolkit/` 或 `scripts/`**
- 不再允许脚本里出现 `/Users/dong/...`

## 3. 最优先要替换的硬编码路径

### A. 原始游戏资源目录
最常见：
- `/Users/dong/Projects/Koukai2`

建议统一替换为：
- CLI 参数：`--raw-dir`
- 或环境变量：`UW2_RAW_DIR`
- 或集中配置文件：`toolkit.toml` / `config.json`

优先级最高脚本：
- `inventory_lzw.py`
- `extract_dat_phase1.py`
- `decode_text.py`
- `extract_messages.py`
- `render_portmap_v2.py`
- `render_font_v1.py`
- `disasm_exe.py`
- `scan_ls11_in_exe.py`
- `decode_eventcg.py`

### B. 原作者研究仓输出目录
常见于 `experiments/`：
- `/Users/dong/Projects/sea2_demo/output/...`

这类脚本优先级较低，因为本来就不是生产工具。建议：
- 直接标记为 `research-only`
- 后续若真需要再迁移

### C. DOSBox 配置 / 调试环境路径
例如：
- `dbg_driver.py`
- `dosbox_game_keys.py`

这类更偏个人环境自动化，不建议进入最小工具包 v1。

## 6. 每类脚本怎么改

### 解压脚本
目标：
- 输入路径改成参数 / 配置
- 输出目录改成 repo-relative
- 提供 CLI 命令

最低改造要求：
- `python ls11_decode.py input.lzw out_dir`
- 不依赖外部状态

### 渲染脚本
目标：
- 输入 part 目录和辅助文件路径都可配置
- 输出统一写到 `output/`
- 允许单文件 / 批量渲染

最低改造要求：
- 港口图、世界图、头像、道具都能单独重跑

### 数据提取脚本
目标：
- 每个提取器单一职责
- 输出 JSON / CSV 格式稳定
- 语义不确定的字段也保留 raw 层

最低改造要求：
- `za_dat`
- `windcur_dat`
- `message/text`

### 反汇编 / CG 脚本
目标：
- 从“作者本机研究笔记”变成“可重复实验”
- 但不要急着承诺稳定 API

最低改造要求：
- 输入 EXE、DAT 路径参数化
- README 清楚写明“研究状态 / 依赖 / 风险”

## 7. 最推荐的配置方式

建议不要每个脚本都单独发明路径参数，最好统一一个配置入口。

### 方案 A：环境变量
```bash
export UW2_RAW_DIR=/path/to/Koukai2
export UW2_OUT_DIR=/path/to/output
```

优点：改动小。

### 方案 B：统一配置模块
例如：
```python
from toolkit.config import RAW_DIR, OUT_DIR
```

优点：以后演进成真正工具包更顺。

### 结论
- **短期**：环境变量最快
- **中期**：统一配置模块最好

## 8. 迁移完成的判断标准

一个脚本不算“迁好了”，除非满足：

1. 没有作者本机绝对路径
2. 输入来源清楚
3. 输出位置清楚
4. 依赖清楚
5. 能在你的机器上从零重跑
6. 失败时能看懂错误，不靠上下文猜

## 10. 一句话结论

你现在最不缺的是“资料”，最缺的是把这些资料的**生成方法**从作者本机环境里拆出来。真正第一优先不是继续导入更多内容，而是：

- 消灭绝对路径
- 固定原始输入目录
- 固定输出目录
- 先让 5~8 个核心脚本稳定重跑
