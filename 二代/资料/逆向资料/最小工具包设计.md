# 最小工具包设计

> 目标不是把 `sea2_demo` 整仓照搬成你的开发基础，而是从中提炼一个**能持续出数据、能支撑复刻主链、复杂度可控**的最小工具包。

## 1. 工具包目标

最小工具包（MVP）只做三件事：

1. **把原始资源解出来**
2. **把关键图像和表跑出来**
3. **把结果稳定输出成你自己的复刻输入格式**

也就是说，它优先服务：
- 世界层
- 港口层
- 基础资源层

而不是一上来覆盖所有剧情 / 反汇编 / CG 研究。

## 2. 工具包边界

### 工具包 v1 应该包含
- LS11 / LS10 / Ls12 解压
- LZW 批量拆包
- Portchip 渲染
- Portmap 渲染
- Worldmap 渲染
- 基础 DAT 数值提取
- 基础文本解码

### 工具包 v1 不应该强塞
- Event CG 完整解码
- SNDT 深度反汇编
- 剧情拓扑分析全链路
- DOSBox 自动化调试
- experiments 历史脚本

## 4. 推荐模块划分

### `toolkit.config`
职责：
- 原始输入目录
- 输出目录
- 字模 / EXE / Chip_no 等辅助文件路径

建议提供：
- 环境变量读取
- 默认 repo-relative 目录
- 参数覆盖

### `toolkit.ls11`
来源脚本：
- `ls11_decode.py`

职责：
- 解码 LS11 / LS10 / Ls12
- 返回 parts 列表

这是整个工具包最底层核心模块。

### `toolkit.render.portchip`
来源脚本：
- `render_portchip_v2.py`

职责：
- 解码 16×16 / 4bpp / 16 色 tile atlas
- 输出 atlas 或 tile 数组

### `toolkit.render.portmap`
来源脚本：
- `render_portmap_v2.py`

职责：
- 把 Portmap 索引和 Portchip atlas 拼成真实港口图
- 支持单港口和全部港口渲染

### `toolkit.render.worldmap`
来源脚本：
- `render_worldmap_v1.py`
- `render_worldmap_v2.py`

职责：
- 解码世界地图 block 结构
- 应用 v2 后处理
- 输出：缩略图、细节图、tile grid

### `toolkit.dat_extract`
来源脚本：
- `extract_dat_phase1.py`
- `extract_messages.py`
- `extract_remaining_dats.py`

职责：
- 提取 `za_dat`
- 提取 `windcur_dat`
- 提取 `monster_dat`
- 后续扩展更多 `.dat`

### `toolkit.text`
来源脚本：
- `decode_text.py`

职责：
- 加载 `KoeiCht.txt`
- 解码 `Message.dat` / `Snr*.mes`
- 对外暴露统一文本解码接口

## 5. CLI 设计建议

建议未来统一一个命令行入口，例如：

```bash
uw2tool ls11-decode raw/Kao.lzw output/lzw_parts/Kao
uw2tool inventory-lzw
uw2tool render-portchip
uw2tool render-portmap
uw2tool render-worldmap
uw2tool extract-dat phase1
uw2tool decode-text raw/Message.dat
```

### 最小 CLI 集

#### A. 解压
- `uw2tool ls11-decode <file> <outdir>`
- `uw2tool inventory-lzw`

#### B. 图像
- `uw2tool render-portchip`
- `uw2tool render-portmap`
- `uw2tool render-worldmap`

#### C. 数据
- `uw2tool extract-phase1`
- `uw2tool decode-text <file>`
- `uw2tool doctor`

## 6. 工具包 v1 的输入 / 输出契约

### 输入契约
要求用户准备：
- `Main.exe`
- `Chip_no.dat`
- `KoeiCht.txt`
- 目标 `.lzw` / `.dat` 文件

全部放入：
- `raw/Koukai2/`

### 输出契约
统一输出到：
- `output/`

例如：
- `output/lzw_parts/`
- `output/portchip_v2/`
- `output/portmap_v2/`
- `output/worldmap_v2/`
- `output/game_data/`

## 7. 建议优先迁入工具包的脚本

第一批直接纳入：
- `ls11_decode.py`
- `inventory_lzw.py`
- `render_portchip_v2.py`
- `render_portmap_v2.py`
- `render_worldmap_v1.py`
- `render_worldmap_v2.py`
- `extract_dat_phase1.py`
- `decode_text.py`

第二批再考虑：
- `extract_messages.py`
- `extract_ships.py`
- `render_font_v1.py`

第三批保留在 research：
- `decode_eventcg.py`
- `decode_eventcg_vram.py`
- `sndt_disasm.py`
- `analyze_*`
- `export_*`

## 8. 为什么不要一开始纳入 event CG

原因不是它不重要，而是它**还不够稳定**：
- 依赖 `unicorn`
- 还带反汇编研究性质
- 更像实验室脚本，不像产线工具

所以更合理的方式是：
- 工具包 v1 不包含
- research 子目录保留
- 等格式和接口稳定后再升级为 v2 模块

## 9. 推荐的里程碑

### Milestone A：生成链跑通
能完成：
- 全 LZW 拆包
- 港口图渲染
- 世界图渲染
- phase1 数据表提取

### Milestone B：文本链跑通
能完成：
- 文本解码
- Message / Snr 文件输出

### Milestone C：复刻输入格式稳定
能产出你自己定义的：
- `ports_for_remake.json`
- `wind_grid.json`
- `world_tiles.json`
- `text_catalog.json`

## 10. 最小工具包 v1 的成功标准

如果下面这些都能做到，就算 v1 成功：

1. 新机器按文档装依赖后可运行
2. `uw2tool doctor` 能准确提示缺什么文件 / 依赖
3. 不依赖作者本机绝对路径
4. 能从原始资源重建核心图像与数据
5. 输出结构稳定、可供复刻工程消费
6. 出错时定位明确，不靠研究上下文猜

## 11. 和当前资料库的关系

你现在这个仓库更像：
- **资料库 + 来源层 + 研究档案馆**

而未来的最小工具包应该是：
- **能重复产出主数据的生产工具链**

两者不冲突，但不要混成一个概念。

## 13. 一句话结论

最小工具包不是把 `sea2_demo` 原样搬走，而是从它里面提炼出一条稳定的“**解压 → 渲染 → 提取 → 规范化**”主链；先把这条主链做好，复刻工程就有了真正可持续的数据工厂。
